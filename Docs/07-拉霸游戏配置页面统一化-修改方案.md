# 拉霸游戏配置页面统一化 - 修改方案

本文档针对后台「系统设置 → 游戏配置」（路由 `/Game/GameConfig`，控制器 `GameConfigController`，视图 `ConfigEditor.cshtml`）中**拉霸类游戏（GameType=3）操作页面与其他子游戏不一致**的问题，给出统一的修改方案，含现状差异、改造目标、对齐范围决策、分项改动清单、数据模型方案、热更新策略与风险回滚。

> 文档定位：方案与实施依据，对应代码 `TTY.Web/Areas/Game/Controllers/GameConfigController.cs`、`TTY.Web/Areas/Game/Views/GameConfig/ConfigEditor.cshtml`、`YYT.BLL/EF/shit/B_LabaGamePara.cs`、`YYT.Entity/EF/M_GameConfigLaba.cs`、表 `gameconfiglaba` / `roomtableconfig`。续接 `02-动态桌台机台配置-实现进度.md`，编号沿用其章节序号风格。

---

## 一、现状：拉霸 vs 其他子游戏的差异清单

「其他子游戏」指押注类(0)、牌机类(1)、鱼机类(2)，三者已统一为「左侧桌台列表 + 新建/删除桌台 + 分区表单」的多桌台独立配置流。拉霸(3) 在前后端均被特殊化分支绕过，差异如下：

| 维度 | 押注/牌机/鱼机 (0/1/2) | 拉霸 (3) | 差异点 |
| --- | --- | --- | --- |
| 数据模型 | 一游戏多桌台：房间表(`parabetroom`/`pararoom`)+机台表(`parabet`/`paracard`/`parafish`)，`ID=gameId*1000+序号` | 一游戏单配置：`gameconfiglaba` 键值表(`GameId,OptKey` 主键)，桌名另存 `roomtableconfig`(由 TC 命令写入) | 拉霸无「桌台行」概念 |
| 左侧桌台列表 | `GetTableConfig` 返回多行，逐桌台渲染 | `GetTableConfig` 仅返回单行 `id=gameId`，列表实为单元素 | 拉霸无多桌台 |
| 新建桌台 | `gcNewTable()` 预填最近桌台参数、桌名留空、`ID=-1` | `gcNewTable()` 首行 `if (type === 3) return;` 直接拦截 | 拉霸不可新建 |
| 新建按钮 | `$('#gcNewTable').toggle(type !== 3)` 显示 | type=3 时隐藏 | 拉霸无新建入口 |
| 删除桌台 | `DeleteTableConfig` 物理删除房间+机台并 `RepackRoomsAfterDelete` 重排 | `DeleteTableConfig` 对 type=3 返回「该类型不支持删除桌台！」 | 拉霸不可删除 |
| 表单分区 | 基础信息 / 下注与兑换 / 类型专属机台设定 / 玩法与踢出 | 仅「基础信息(桌名+开关)」+「拉霸参数(玩家赢概率/金币兑换率)」，`.gc-common` 整体隐藏 | 拉霸缺下注兑换、玩法踢出分区 |
| 类型可见性控制 | `gcApplyType()`：`.gc-only-0/1/2` 按类型显隐，`.gc-common` 显示 | `gcApplyType()`：type=3 走 `else` 分支，`.gc-common` 全部隐藏、仅显 `.gc-base-info`+`.gc-only-3` | 拉霸走独立显隐分支 |
| 保存逻辑 | `SaveTableConfig` 走 `B_BetGamePara/B_CardGamePara/B_FishGamePara.SaveTableFull`，含 `AllocTableId`、`ROOM_MAX` 同步 | `SaveTableConfig` 走 `B_LabaGamePara.SaveRoomPara(labaList, labaTableName)`，按键名(`ExchangeScore`/`AIWinLuckyAtA2`/`PlayerWin`)更新 | 拉霸走独立保存路径 |
| 保存按钮文案 | 新建态「新建桌台」/更新态「更新桌台」 | 固定「保存并热更新」 | 文案不统一 |
| 表单字段 | 桌名、开关、最小/最大下注、兑换单位、兑换币值、兑换金币数、入场金币、无发炮踢出时间/开关 + 类型专属 | 桌名、开关、玩家赢概率、金币兑换率 | 拉霸字段最少 |
| 字段校验 | 下注非负、最小≤最大、踢出时间非负等 | 仅赢概率 0-10、兑换率≥1 | 拉霸校验最弱 |

**根因**：拉霸在游戏设计上本就是「单桌台/单配置」游戏（`B_LabaGamePara.GetGameParams` 返回单游戏键值集合，无桌台维度），而页面改造时为最小化风险，对 type=3 做了前端特殊化绕行，导致其操作体验与其他三类割裂。

---

## 二、改造目标

1. **操作体验统一**：拉霸在「游戏配置」页的操作流程（选择游戏 → 加载桌台列表 → 选中/新建/删除 → 编辑分区表单 → 保存并热更新）与押注/牌机/鱼机一致，去除前端 `type === 3` 的特殊化分支。
2. **表单分区统一**：拉霸使用与其他子游戏相同的分区结构（基础信息 / 下注与兑换 / 拉霸专属机台设定 / 玩法与踢出），不再只显示两个分区。
3. **交互统一**：新建/删除桌台、保存按钮文案、字段校验、空态占位、重置归零等行为对拉霸同样生效。
4. **底层最小侵入**：优先复用既有 `gameconfiglaba` 键值模型与 `B_LabaGamePara` 保存路径，避免一次性推翻拉霸数据模型带来的 C++ 侧读取风险。

---

## 三、关键决策：对齐范围（需确认）

拉霸「改成和其他子游戏一样的操作页面」存在两种理解，决定方案是「轻量 UI 统一」还是「多桌台架构改造」，**实施前需确认**：

### 方案 A：完全对齐（多桌台独立配置）—— 推荐
拉霸也支持「一游戏多桌台」：左侧桌台列表、新建/删除桌台、每桌台独立配置，与鱼机/牌机/押注完全同构。

- **优点**：彻底消除特殊化，体验完全一致；符合 RTF「每张桌台都可独立配置」的统一方向；未来新增拉霸桌台无需再改前端。
- **代价**：需扩展 `gameconfiglaba` 增加「桌台维度」（见第五章方案①），改造 `B_LabaGamePara` 与控制器分流；C++ 侧 RP 读取路径基本不动（按 `GameId+TableIndex` 取键值）。
- **适用**：拉霸未来确实可能开多桌（如不同赔率档位的拉霸桌）。

### 方案 B：仅 UI/交互对齐（单桌台）
拉霸仍保持「单配置/游戏」，但表单分区、保存/重置/校验交互与其他子游戏统一，仅去除前端 `type===3` 特殊化分支。

- **优点**：改动最轻，不碰数据模型，C++ 侧零改动；快速消除体验割裂。
- **代价**：拉霸仍无「新建/删除桌台」「左侧多桌台列表」（列表恒为单元素），与「完全一样」仍有差距；未来要开多桌仍需回头做方案 A。
- **适用**：拉霸确定永远只单桌台，只求表单观感与操作一致。

> **本文后续以「方案 A（完全对齐，推荐）」为主线展开**；方案 B 的差异在第十章单独列出，便于按确认结果取舍。

## 四、数据模型决策（仅方案 A 需确认）

若选方案 A，拉霸多桌台参数存储有三种候选，**实施前需确认**：

### 候选①：扩展 `gameconfiglaba`（推荐）
给 `gameconfiglaba` 增加 `TableIndex`（或 `TableId`）维度，键值模型不变，按桌台分键存储。

- 主键由 `(GameId, OptKey)` 改为 `(GameId, TableIndex, OptKey)`；
- 桌名仍由 TC 命令写入 `roomtableconfig`（既有路径不动）；
- `B_LabaGamePara` 按 `GameId+TableIndex` 过滤读写；
- C++ 侧 RP 读取路径基本不动（仅多带一个 TableIndex 维度）。

**优点**：对拉霸既有逻辑改动最小，键值模型天然适配拉霸「赔付档位 Payout0..N / 房间参数 betMin/betMax...」的异构参数集；不动 `roomtableconfig`、`pararoom`。
**代价**：`gameconfiglaba` 主键迁移 + 历史数据回填 `TableIndex=0`。

### 候选②：迁入 `roomtableconfig`
把拉霸桌台参数迁入 `roomtableconfig`（C++ TC 命令已读写的统一动态桌台表），符合项目「统一桌台配置」方向。

- 拉霸赔率档位等异构参数与 `roomtableconfig` 的固定列结构不匹配，需额外列或 JSON 扩展列；
- C++ 侧需调整拉霸参数读取来源（从 `gameconfiglaba` 切到 `roomtableconfig`），改动面大、风险高。

**优点**：长期最统一。
**代价**：表结构不匹配拉霸异构参数；C++ 侧读取来源切换，回归风险高。**不推荐**。

### 候选③：复用 `pararoom`
让拉霸复用 `pararoom`（房间表）+ 为拉霸留空机台表，最「同质」于牌机/鱼机。

- 拉霸无传统机台概念，`parafish/paracard` 字段（DIF/HYPE_TYPE 等）与拉霸赔率/赢概率参数形态差异大，需大量适配；
- 拉霸赔率档位是变长键值（Payout0..9 视符号数而定），塞进 `pararoom` 固定列不自然。

**优点**：与牌机/鱼机数据流最一致（`ID=gameId*1000+序号`、`ROOM_MAX` 同步直接复用）。
**代价**：参数形态差异大，适配成本高，且需 C++ 侧为拉霸单独解释 `pararoom` 字段。**不推荐**。

> **推荐候选①**：在最小侵入前提下获得多桌台能力，C++ 侧 RP 路径不变，仅多带 TableIndex 维度。

## 五、改造内容（方案 A 主线）

### 1. 数据库迁移 — `gameconfiglaba` 增加桌台维度
新增迁移脚本 `Docs/sql/拉霸多桌台配置-字段扩展.sql`（沿用本项目「独立迁移脚本，不改 `mth.sql` 全量字典」的约定，见 `02` 文档二.2）：

```sql
-- 1) 增加 TableIndex 列，默认 0（兼容现有单桌台数据）
ALTER TABLE `gameconfiglaba`
  ADD COLUMN `TableIndex` int(11) NOT NULL DEFAULT 0 COMMENT '桌台索引(0..N-1)' AFTER `GameId`;

-- 2) 主键调整为 (GameId, TableIndex, OptKey)
ALTER TABLE `gameconfiglaba` DROP PRIMARY KEY,
  ADD PRIMARY KEY (`GameId`, `TableIndex`, `OptKey`) USING BTREE;

-- 3) 历史数据回填 TableIndex=0（拉霸原本单桌台）
--    无需显式 UPDATE，DEFAULT 0 已覆盖；此处仅作说明。
```

约定（与 `roomtableconfig` 一致）：`TableIndex = 桌台ID % 1000`，`桌台ID = gameId*1000 + TableIndex`，与押注/牌机/鱼机的 `ID` 编址规则对齐，便于 `AllocTableId`/`RepackRoomsAfterDelete` 复用。

### 2. 实体扩展 — `YYT.Entity/EF/M_GameConfigLaba.cs`
新增 `TableIndex` 属性，主键顺序调整为 `(GameId, TableIndex, OptKey)`：

```csharp
[Key, Column(Order = 1)]
public int GameId { get; set; }

[Key, Column(Order = 2)]
public int TableIndex { get; set; }   // 新增：桌台索引

[Key, Column(Order = 3)]
public string OptKey { get; set; }
// 其余不变
```

### 3. BLL 改造 — `YYT.BLL/EF/shit/B_LabaGamePara.cs`
新增 `SaveTableFull` 入口（与 `B_CardGamePara/B_FishGamePara.SaveTableFull` 同名同义），统一多桌台保存 + 热更新：

- `GetGameParams(gameId)` → 按 `(gameId, tableIndex)` 过滤返回该桌台键值集合；提供 `GetTableList(gameId)` 返回该游戏所有桌台索引列表（供控制器渲染左侧列表）。
- `SaveTableFull(int tableId, int gameId, List<M_GameConfigLaba> paras, string tableName)`：
  - `tableIndex = tableId % 1000`；
  - 事务内 upsert 该 `(gameId, tableIndex)` 下全部键值（赔率档位、betMin/betMax/coinsNeed、赢概率、兑换率等）；
  - Commit 后发 `SConnect.SendReadString(EScMsgType.RP, gameId)` 强校验热更新（沿用既有 `SaveRoomPara` 的 RP 策略）；
  - 桌名非空时追加 `SendTcCommand(gameId, 0, tableIndex, tableName, enabled, idleFireTimeoutSec, idleFireKickEnabled, maxSeats)` 写 `roomtableconfig`（沿用既有 TC 路径，仅 `tableIndex` 由 0 改为实际偏移）；
  - 返回 `Msg`：`code==1` 成功；`code==0 && datas==true` 表示落库成功但热更失败（与 `B_SuperPara.PushHotUpdate` 的 `datas=true` 约定一致，见 `02` 文档三.9）。
- 保留 `SaveRoomPara` 旧签名供旧路径兼容（标记 `[Obsolete]`）。
- 新增 `DeleteTable(int tableId, int gameId)`：删除 `(gameId, tableIndex)` 全部键值，重排剩余桌台 `TableIndex` 为连续 0..k-1（复用 `GameConfigController.RepackRoomsAfterDelete` 的「负数临时 id 两段式重排」思路，避免主键碰撞）。

> 注意：拉霸无 `paragame.ROOM_MAX` 概念（不走 `pararoom`），`SyncRoomMaxToRoomCount` 不适用于拉霸；服务端按 RP 全量重载 `gameconfiglaba`，故无需同步 `ROOM_MAX`。这点与押注/牌机/鱼机不同，需在控制器分流时跳过。

### 4. 控制器改造 — `GameConfigController.cs`
目标：去除 `gameType==3` 的特殊化分支，让拉霸走与其他类型同构的「列表 + 新建/删除 + 保存」流。

- **`GetTableConfig`**：`else` 分支（type=3）改为返回多行——调用 `B_LabaGamePara.GetTableList(gameId)`，逐桌台组装行对象（`id=gameId*1000+tableIndex`、`tableName`、`enabled`、`winRate`、`exchangeRate`、`betMin`、`betMax`、`coinsNeed` 等），与其他类型行对象字段对齐。
- **`SaveTableConfig`**：`if (gameType == 3)` 分支改为收集完整表单（含下注/兑换/玩法踢出），组装 `List<M_GameConfigLaba>`（每个可配项一条，`TableIndex=tableId%1000`），调用 `B_LabaGamePara.SaveTableFull(tableId, gameId, list, tableName)`；`isNew` 判定与 `AllocTableId` 对拉霸生效。
- **`DeleteTableConfig`**：`else` 分支去掉「该类型不支持删除桌台」，改为 `gameType==3` 时调用 `B_LabaGamePara.DeleteTable(tableId, gameId)`；拉霸无 `RepackRoomsAfterDelete`（无 `pararoom`），由 BLL 内部重排 `TableIndex`。
- **`AllocTableId`**：拉霸改为查 `gameconfiglaba` 的 `MAX(TableIndex)`，`next = gameId*1000 + maxTableIndex + 1`。
- **校验**：拉霸保存前补齐与其他类型一致的校验（下注非负、最小≤最大、踢出时间非负），统一在控制器层完成。

### 5. 视图改造 — `ConfigEditor.cshtml`
去除前端 `type === 3` 特殊化，统一分区与交互：

- **`gcApplyType()`**：删除 type=3 的 `else` 分支，改为与 0/1/2 一致——`.gc-common` 全部显示，按类型显隐 `.gc-only-3`（拉霸专属：玩家赢概率、金币兑换率、赔率档位 Payout0..N）。
- **`gcNewTable()`**：删除首行 `if (type === 3) return;`，拉霸同样支持「预填最近桌台参数、桌名留空、`ID=-1`」新建态。
- **新建按钮**：删除 `$('#gcNewTable').toggle(type !== 3)`，拉霸同样显示新建入口。
- **`gcUpdateSaveBtn()`**：删除 `if (gcGetType() === 3) { ... return; }`，拉霸同样按 `ID` 新建/更新态切换文案。
- **`gcSave()`**：删除 `if (type === 3) { ...独立序列化分支... return; }`，拉霸走统一序列化 + 校验 + 提交路径（表单字段名与控制器对齐）。
- **`gcFill()` / `gcZeroForm()`**：拉霸分支补齐下注/兑换/玩法踢出字段的回填与归零，与其他类型一致。
- **表单分区补全**：拉霸 `.gc-only-3` 区块在「玩家赢概率/金币兑换率」基础上，新增「赔率档位」（Payout0..N，按游戏符号数动态渲染）、「下注与兑换」分区共用字段（最小/最大下注、入场金币）等，使分区结构与 0/1/2 对齐。
- **`gcLoadTables()`**：删除 `else if (gameType === 3) { gcSelectTableById(GC_TABLES[0].id); }`，改为与其他类型一致的「加载后默认进入预填新建态或选中首桌」。

## 七、热更新与一致性策略

1. **保存后强校验**：拉霸保存走 `B_LabaGamePara.SaveTableFull`，Commit 后发 `EScMsgType.RP`，`code==1` 才算成功；否则 `code=0 && datas=true`（已落库标记）+ 明确文案，**绝不静默失败**（与 `02` 文档二.4、三.9 的策略一致）。
2. **桌名热更新**：`tableName` 非空时追加 `SendTcCommand`，`tableIndex` 由 0 改为实际偏移；TC 失败仅追加提示，不影响落库结果（沿用 `B_LabaGamePara.SaveRoomPara` 既有 TC 容错）。
3. **前端刷新**：`gcSave` 以 `data.code==1 || data.datas===true` 触发 `gcLoadTables`（从 DB 读取，与热更新状态无关），与 `02` 文档三.9 修复一致。
4. **删除热更新**：拉霸删除后发 `RP` 全量重载；`TableIndex` 重排由 BLL 在事务内完成，保证服务端重载后桌台列表连续。
5. **`ROOM_MAX` 不适用**：拉霸不走 `pararoom`，`SyncRoomMaxToRoomCount` 对拉霸跳过（控制器分流时判断）。

## 九、风险与回滚

| 风险 | 影响 | 缓解/回滚 |
| --- | --- | --- |
| `gameconfiglaba` 主键迁移破坏既有 C++ 读取 | 拉霸参数读不到，游戏异常 | 迁移脚本 `DEFAULT 0` 保证旧数据 `TableIndex=0` 可读；C++ 侧仅需多带 TableIndex 维度，不传则等价旧逻辑。回滚：`DROP COLUMN TableIndex` + 还原主键 |
| 拉霸多桌台与 C++ 服务端 `RP` 全量重载不兼容 | 新桌台不生效 | 上线前在测试服验证 `RP` 后服务端能加载多 `TableIndex`；不行则回退方案 B（单桌台） |
| 前端去特殊化后拉霸字段缺失/错位 | 表单空白或保存报错 | `gcFill/gcZeroForm` 拉霸分支逐字段对齐控制器返回结构；联调覆盖新建/编辑/删除三态 |
| 赔率档位动态渲染（水浒传 10 档 vs 其他 8 档） | 档位输入数量不对 | 按 `games` 表符号数动态生成 `PayoutN` 输入框，前端从 `GetGameList` 携带符号数 |
| 删除重排 `TableIndex` 与服务端缓存不一致 | 桌台错位 | 重排后强制发 `RP` 全量重载；与 `RepackRoomsAfterDelete` 同策略 |

## 十、方案 B 差异（仅 UI/交互对齐，单桌台）

若确认选方案 B，则**不碰数据模型**，仅做前端与控制器交互统一：

- **不改**：`gameconfiglaba` 表、`M_GameConfigLaba`、`B_LabaGamePara.SaveRoomPara` 签名、`roomtableconfig` 写入逻辑。
- **视图**：`gcApplyType()` 删 type=3 的 `else`，`.gc-common` 显示、`.gc-only-3` 显隐；补「下注与兑换」「玩法与踢出」分区字段（下注/入场金币等映射到 `gameconfiglaba` 的 `betMin/betMax/coinsNeed` 键，已有数据）。
- **视图**：`gcNewTable()` 删 `if (type === 3) return;`，但拉霸单桌台场景下「新建」语义退化为「重置当前桌台参数为默认」（列表恒单元素，`ID=gameId` 不变），需在文案上区分（如「重置参数」而非「新建桌台」）；或保留隐藏新建按钮、仅统一表单分区。
- **控制器**：`SaveTableConfig` type=3 分支补齐下注/兑换字段收集（写 `betMin/betMax/coinsNeed` 键），其余沿用 `SaveRoomPara`。
- **删除**：拉霸单桌台不支持删除，`DeleteTableConfig` 保留「该类型不支持删除」提示。
- **代价**：拉霸无多桌台，与「完全一样」仍有差距；但表单观感与保存/校验交互统一。
