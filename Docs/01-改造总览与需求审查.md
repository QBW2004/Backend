# 5.23优化版后台改造审查总结

本文基于只读审查：`5.23优化版.rtf`、`winner.sql`、`Backend` 后台代码、`Server` 服务端协议与现有文档。目标是为下一步改造后台代码提供入口、范围和风险清单。

## 一、RTF核心需求提取

### 1. 动态桌台/机台配置

需求：
- 大厅取消“初级/中级/高级”房间分组展示，进入游戏后直接展示后台已开启的桌台/机台。
- 后台新增或修改桌台后，前台应及时看到，不再要求重启服务器或重登客户端。
- 鱼机、牌机、单挑等每张桌台都可独立配置：桌名、最小下注、最大下注、兑换比率、带入金币、难度、无发炮踢出时间或不踢出。
- 桌台坐席扩容：游戏内桌台支持从原有的 4 人一桌扩容至 6 人一桌。
- 金鲨银鲨、奔驰宝马可暂时保持现状。

后台现状：
- 已有游戏参数页：`TTY.Web/Areas/Game/Controllers/GameParaController.cs`。
- 参数表已有基础分层：
  - `games`：游戏开关、游戏类型。
  - `pararoom`：鱼机/牌机房间参数，含 `BET_MIN/BET_MAX/EX_COIN/COIN_SC/COIN_NEED/scoreSwitch`。
  - `parafish`：鱼机机台参数，含 `DIF/HAR/SITE_TYPE`。
  - `paracard`：牌机机台参数，含 `DIF/HYPE_TYPE`。
  - `parabetroom/parabet`：押注类房间与机台参数。
- 后台远程命令已有：
  - `PA`：更新机台参数。
  - `RP`：房间参数刷新。
  - `PR`：鱼机房间参数刷新。
  - `GA/CA`：库存查询/清空。
- `SaveRoomBetConfig()` 已有保存后通知服务端 `EScMsgType.RP` 的逻辑。

主要缺口：
- 当前表结构没有“桌名”“无发炮踢出秒数/是否踢出”等字段。
- 当前参数模型仍偏“房间+机台模板”，RTF 要求更接近“每张桌台独立配置”。
- 桌台参数缺少关于最大坐席数的配置，若写死会导致扩容至 6 人时产生异常。
- 保存配置后是否所有游戏都能热更新，需要逐游戏验证；不能只看后台写库成功。

建议后台改造：
- 新增或扩展桌台配置字段，优先以兼容旧结构方式扩展：
  - `TableName`
  - `MaxSeats` (支持扩容配置，例如设为 6)
  - `IdleFireTimeoutSec`
  - `IdleFireKickEnabled`
  - `Enabled`
  - 如需要支持小数炮值，金币/炮值字段建议改用“整数分单位”而不是数据库浮点，例如 `MinBetUnits=1` 表示 0.1。
- 后台页面按游戏类型统一抽象“桌台配置编辑器”，但保存仍走现有 `B_FishGamePara/B_CardGamePara/B_BetGamePara` 分支，降低一次性重构风险。
- 每次保存配置后必须明确执行对应远程命令，并把失败提示返回页面：“保存成功但服务端未热更新”不能静默。

### 2. 鱼机炮值与难度

需求：
- 炮值支持 0.1 到 1 这种小数起步，点击加炮按最低值递增。
- 鱼机难度支持 0.5 粒度，例如 1.5 表示 15%。
- 小倍率鱼更容易死亡，大倍率鱼极低概率，大鱼不能频繁爆。
- 小鱼数量增加，1-10倍鱼在当前基础上增加约 0.5-0.8 倍。

后台现状：
- `pararoom.BET_MIN/BET_MAX/scoreSwitch` 多为整数。
- `parafish.DIF/HAR/SITE_TYPE` 当前是整数难度/控制参数。
- `GameParaController.SaveConfig()` 读取表单后进入对应 BLL 保存。

主要缺口：
- 参数精度不满足 0.1 炮值、0.5 难度。
- 出鱼数量属于子游戏算法/配置，不只后台表单改字段。

建议后台改造：
- 页面允许输入小数，但入库建议转成整数单位，避免 C++ 协议与 MySQL 浮点误差。
- 明确“显示值”和“协议值”的换算，例如后台显示 0.1，入库 `1`，客户端/服务端按 `scale=10` 解释。
- 鱼倍率分段策略不应只放在后台，后台只负责可配置参数；最终死亡率、出鱼量要服务端算法落地。

### 3. 牌机出货、库存和新增游戏

需求：
- 牌机包括火凤凰、大字板、金皇冠、NBA，后续 ATT、水浒、明星97、水果机。
- 每台机单独配置牌型倍数、出货概率、库存。
- 牌型示例：杂牌、一对、两对、三条、顺子、同花、葫芦、四梅、同花小顺、同花大顺、五梅。
- 后台展示牌型倍数，方便判断送出多少分。

后台现状：
- `paracard.DIF` 是字符串形式，例如 `3434453555777533`，表达能力较弱且不透明。
- `paragame`、牌机库存结构存在，但需要进一步核对具体算法绑定。
- `games` 中已有 ATT 系列、明星97、水果机等游戏项，但启用与服务端可运行项目不完全等价。

主要缺口：
- 牌型概率/库存没有清晰结构化表。
- 后台缺“每台机每个牌型”的可视化配置。
- 新增游戏不能只改后台 `games`，必须确认服务端项目、协议、客户端资源都存在。

建议后台改造：
- 新增结构化牌型配置表，例如 `card_payout_profile`：
  - `GameId/TableId/HandType/PayoutMultiplier/ProbabilityBasis/StockLimit/StockRemain/Enabled`
- 后台旧 `DIF` 字符串保留兼容，新表先用于新算法或逐步迁移。
- “ATT 保留一个，其余隐藏”应通过 `games.Enable` 与后台展示过滤实现，但服务端枚举和客户端入口也要同步。

### 4. 玩家控制：点杀、放水、金币上限

需求：
- 鱼机：金币上限、放水、点杀。
- 押注类：单挑、狮子、奔驰宝马、金鲨银鲨支持点杀/放水。
- 牌机：增加点杀吃分，后台“匿名控制”改成“放水”，并拆分权限：牌机控制、牌机放水。
- 控制都要求“慢慢吃/慢慢给”，达到设定金币目标后自动失效。

后台现状：
- `UserInfoController` 已有：
  - `ChgUserRate()` 鱼机控制。
  - `ChgUserRateBet()` 押注控制。
  - `ChgUserRatePAIJI()` 牌机控制。
- 权限字段已有：
  - `admin.IsKill`
  - `admin.IsRelease`
  - `admin.IsProbability`
  - `admin.IsKicking`
- `manageropt` 只记录 `UserID/Opt/OptValue/Type/TIME`，更像一次性控制记录。
- `usercontrolvalue` 已存在，需进一步确认现有代码使用深度。

主要缺口：
- 现有控制偏“概率值/次数”，不是“目标金额、已吃/已放、自动失效”的状态机。
- 权限命名与需求不完全一致，尤其牌机“控制/放水”要拆。
- 后台备注展示不够，代理无法直观看到某玩家当前控制状态、目标、进度。

建议后台改造：
- 新增玩家控制状态表，避免把金额状态硬塞到 `manageropt`：
  - `UserID`
  - `GameType/GameId`
  - `ControlMode`：Kill/Release/Limit
  - `TargetCoins`
  - `ConsumedCoins`
  - `GrantedCoins`
  - `LimitCoins`
  - `Status`
  - `CreatedBy/CreatedTime/ExpiredTime`
- `manageropt` 继续作为审计日志，不作为当前状态唯一来源。
- 后台控制提交后仍走 `SConnect.SendReadString(EScMsgType.UC, ...)`，但命令内容需要支持扩展参数，否则只能传旧格式。

### 5. 在线用户实时监管

需求：
- 在管理后台新增或强化“在线用户”，只显示进入游戏内的玩家，不显示仅在大厅的玩家。
- 字段：用户账号、所属代理、游戏名称、桌号、押注金额、备注、余额、游戏分、总充值、总兑换、总盈亏、控制、充币、扣币。
- 支持实时刷新，方便代理操作。

后台现状：
- `UserInfoController.GetOnlinePlayers()` 已存在。
- `B_Users.GetOnlinePlayersList()` 已存在。
- `M_Users_DTO` 已有实时字段：
  - `CurrentGameName`
  - `CurrentGameType`
  - `CurrentRoomId`
  - `CurrentTableId`
  - `CurrentSeatId`
  - `CurrentBetCoins`
  - `CurrentWinLose`
- `FillCurrentPlayInfo()` 会填充当前游戏/桌台/下注/输赢。

主要缺口：
- 实时数据来源需要确认是否来自服务端持续上报，还是查询旧日志。
- “只显示游戏内，不显示大厅”当前用 `INHALL` 可能语义不够精确。
- 缺控制备注、控制进度、游戏分、总充值/总兑换/总盈亏聚合列。

建议后台改造：
- 优先复用 `M_Users_DTO` 扩字段，不新建一个完全独立列表。
- 在线状态应区分：
  - 是否在线
  - 是否在大厅
  - 是否在房间/桌台
  - 是否在游戏中
- 在线页数据刷新可先用短轮询，SignalR 已引入但不必第一阶段强依赖。
- 增加权限过滤，代理只能看自身管理范围内玩家，沿用 `B_Admin.GetManagedAgencyAccounts()` 这类能力。

### 6. 代理客服链接

需求：
- 每个代理独立客服链接。
- 玩家用某代理邀请码注册后，前台客服跳到该代理链接。

后台现状：
- `admin.CustomerServiceUrl` 已存在。
- `CustomerServiceUrlController`、`OnlineServiceController` 存在。
- `Database.Register()` 注册时通过 `Admin.InviteCode` 找到 `AGENCY`。

建议后台改造：
- 客服链接读取逻辑统一按玩家 `Users.AGENCY -> Admin.CustomerServiceUrl`。
- 若下级代理为空，向上查找或使用总后台默认链接，规则要固定。
- 后台代理管理页补充客服链接编辑和权限约束。

### 7. 异常断线与数据一致性（防卡机吞分）

需求：
- 彻底解决由于玩家卡机、断线、顶号等异常状况导致的数据丢失（金币没有了）问题，确保核心资产强一致性。

后台与架构现状：
- 此问题极易在服务端高并发场景下由于事务处理不当引发，是极其严重的架构缺陷。若完全依赖C++内存或完全依赖MySQL都无法有效防范。

建议后台与服务端联合改造（构建双重防护网）：
- **C++ 服务端职责（状态守护）**：强化心跳检测与断线托管。检测到异常断线时，立刻触发结算回调异步写入数据库。对于顶号登录必须强制执行“互斥锁”三步走机制（踢出A -> 强制结算写入DB并等待确认 -> 允许B读取）。
- **MySQL 数据库职责（最后仲裁）**：金币结算更新必须包裹在完整的事务（`BEGIN ... COMMIT`）中，确保余额表与流水日志表同生共死；**必须使用增量更新**（如 `UPDATE Users SET Gold = Gold + X`）并配合行锁，坚决禁止使用绝对值直接覆盖。
- **极端宕机快照保护**：C++ 服务端应每隔数秒对内存中的临时赢余状态进行快照存储，有条件可增加 Redis 作为高速缓存层，防止进程级崩溃导致玩家本金全毁。

## 二、数据库审查要点

核心表：
- 账号/代理：`admin`、`agent_hierarchy`、`agent_permission_template`、`users`、`userrelations`、`invite_codes`。
- 参数：`games`、`pararoom`、`parafish`、`paracard`、`parabetroom`、`parabet`、`gameconfiglaba`、`gameacclaba`。
- 控制/日志：`manageropt`、`usercontrolvalue`、`useroptlog`、`agencyoptlog`、`tablecoinrecord`。
- 在线/快照：`users.INHALL/GAME_SCORE/BMW_SCORE`、`userscoresnapshotledger`。

注意：
- `winner.sql` 根目录版本和 `Backend/winner.sql`、`Server/winner.sql` 需要在改造前做一致性确认。
- SQL 注释编码有错位，但字段名和数据可读。
- `Web.config` 与 `YYT.DbUtility/PubConstant.cs` 中存在明文数据库账号，后续部署要单独治理。

## 三、后台下一步改造顺序建议

1. 先定数据模型：桌台配置、玩家控制状态、牌型库存配置。
2. 再补后台页面：桌台列表/编辑、在线用户、控制弹窗、代理客服。
3. 再补远程命令：确认 `MsgDefine.config` 与服务端命名管道命令支持新参数。
4. 最后逐游戏联调热更新：保存后台参数后，中心服和子游戏无需重启即可生效。

## 四、重点风险

- 不要只改后台表单，服务端协议和子游戏内存参数不更新，前台仍不会实时生效。
- 小数炮值不要直接用浮点贯穿数据库、C++、客户端协议，优先整数缩放。
- 玩家控制要做成可恢复状态机，否则服务端重启后“吃了多少/放了多少”会丢。
- 在线用户页如果依赖日志而不是当前状态，会出现延迟、重复和误显示大厅用户。
- 牌机出货配置结构化后，要保留旧 `paracard.DIF` 兼容，否则旧游戏可能无法启动。
- **桌台坐席扩容越界风险**：将 4 人桌扩容至 6 人时，不仅是前台客户端修改。后台数据库对应配置表必须增加坐席数上限字段（如 `MaxSeats = 6`），同时**务必强提醒 C++ 服务端开发人员同步修改座位数组长度**（避免因硬编码 `[4]` 导致服务端内存直接越界崩溃）。
- **高并发金币结算与断线吞分隐患**：这是最致命的系统架构风险。不能仅靠单一端解决，必须从底层排查 MySQL 事务处理与游戏服 C++ 内存状态的同步逻辑，严格落实“C++负责状态守护，MySQL负责增量事务”的强一致性机制，严防玩家卡机、顶号导致的核心资产蒸发。
