# 拉霸类后台游戏配置分类型改造方案

> 状态：方案设计 | 日期：2026-07-09
> 涉及：Web 后台「系统设置 -> 游戏配置」拉霸类(gameType==3)分支
> 涉及游戏：水浒传(Shz, gameId=53)、明星97(Mx97, gameId=16)、水果拉霸(Fruit, gameId=40)
> 前提：三款中仅 Shz 可正常游玩;Mx97/Fruit 功能未开发完;三款的桌台参数集尚未最终定稿,本方案提供可扩展的分类型框架

## 一、现状问题


### 1.2 问题清单

| # | 问题 | 说明 |
|---|---|---|
| 1 | **winRate 对三款都是死代码** | Shz(53)/Mx97(16) 不在 winKey 硬编码的 39/40/41 里;Fruit(40) 虽在但服务端不读 PlayerWin;三服全无 winRate 概念 |
| 2 | **exchangeRate 对三款都是死代码** | 三服不读 gameconfiglaba 的 ExchangeScore;兑换率走 oneCoinScore(写死/中心服下发) |
| 3 | **Payout 赔付表无法编辑** | 三服 payoutTable 写死在代码里(Shz 9值/Mx97·Fruit 8值),UI 没有 Payout 输入框,gameconfiglaba 有 Payout0-9 种子数据但无保存路径 |
| 4 | **dif/har 难度只有 Shz 用,UI 不显示** | Shz 有 tablePara[dif/har]+COM_TABLE_SET;Mx97/Fruit 无此机制;当前 UI 无难度字段 |
| 5 | **winKey 硬编码漏掉 16/53** | `gameId==39?...:(gameId==40||gameId==41?...:string.Empty)`,Shz(53)/Mx97(16) 被排除 |
| 6 | **改名因 val>0 门控失效** | B_LabaGamePara.SaveTableFull 的 TC 被 gameconfiglaba 写入门控(详见 Shz桌台改名失效诊断) |
| 7 | **Mx97/Fruit 功能未完成,参数待定** | 玩法/下注参数可能删除或更换,UI 不应固化当前字段 |

## 二、设计原则

因为三款参数**尚未最终定稿**(Shz 待修改、Mx97/Fruit 待开发),方案必须**可扩展**,不能把字段硬编码死:

1. **按子类型分**:拉霸类(gameType==3)内部按 gameId 细分三种子类型,各自有独立字段集
2. **字段集可配置**:子类型的字段定义放配置/常量,新增/删除字段只改配置不动核心逻辑
3. **待定字段标注**:未定稿的字段在方案里标"待定",先预留位置,后续填充
4. **死代码清理**:winRate/exchangeRate 统一移除(三款都不用)
5. **改名先修**:改名(val>0 门控)是 P0,不依赖参数定稿,先修通

## 三、子类型划分

按 gameId 细分(方案 A:硬编码,因拉霸仅 3 款且短期不会新增):

| 子类型 | gameId | 标识 | 状态 | 说明 |
|---|---|---|---|---|
| **LabaShz** | 53 | `labaSubType=3` | ✅ 可玩 | 3×5 九线 + 免费局变王;有 dif/har 难度;9 个赔付值 |
| **LabaMx97** | 16 | `labaSubType=1` | ⚠️ 未完成 | 3×3 拉霸 + 比倍/存取币;8 个赔付值;无难度 |
| **LabaFruit** | 40 | `labaSubType=2` | ⚠️ 未完成 | 3×3 拉霸 + 转轮免费;8 个赔付值;无难度 |
| 默认 | 其他 | `labaSubType=0` | - | 走通用最小集(仅桌名) |

控制器判断:

```csharp
// GameConfigController 内新增辅助方法
private static int GetLabaSubType(int gameId)
{
    if (gameId == 53) return 3;  // 水浒传
    if (gameId == 16) return 1;  // 明星97
    if (gameId == 40) return 2;  // 水果拉霸
    return 0;                     // 其他/未知
}
```

### 5.3 BLL 改动(B_LabaGamePara.cs)

#### 5.3.1 SaveTableFull 修复改名 + 新增 enabled

```csharp
// 签名新增 enabled 参数
public Msg SaveTableFull(int tableId, int gameId, List<M_GameConfigLaba> paras,
                         string tableName, int enabled)
{
    Msg msg = new Msg(0, "保存失败！");
    int tableIndex = tableId % 1000;
    using (var ef = new GameDbContext())
    {
        // ① gameconfiglaba 写入(保持原逻辑,但改为 upsert:找不到就插入)
        var rst = ef.GameConfigLabas.Where(c => c.GameId == gameId && c.TableIndex == tableIndex).ToList();
        foreach (var p in paras)
        {
            var existing = rst.FirstOrDefault(r => r.OptKey == p.OptKey);
            if (existing != null && p.OptValue > -1)
            {
                existing.OptValue = p.OptValue;
                ef.Entry(existing).State = EntityState.Modified;
            }
            else if (p.OptValue > -1)
            {
                // 找不到就插入(修复原 update 空转问题)
                p.GameId = gameId;
                p.TableIndex = tableIndex;
                ef.GameConfigLabas.Add(p);
            }
        }
        int val = ef.SaveChanges();

        // ② RP:仅当 gameconfiglaba 有改动时发
        if (val > 0)
        {
            var srv = new SConnect();
            var tmpMsg = srv.SendReadString(EScMsgType.RP, gameId);
            msg.code = tmpMsg.code;
            msg.content = tmpMsg.content;
        }

        // ③ TC:解耦!只要 tableName 非空就发(不依赖 val>0)
        if (!string.IsNullOrEmpty(tableName))
        {
            try
            {
                var srv2 = new SConnect();
                var tc = srv2.SendTcCommand((ushort)gameId, 0, (ushort)tableIndex,
                    tableName, (byte)enabled, 0u, 0, 6);
                if (tc != null && tc.code == 1)
                {
                    if (msg.code == 0 && val == 0) { msg.code = 1; msg.content = "桌名更新成功"; }
                }
                else
                {
                    msg.datas = true;
                    msg.content = (string.IsNullOrEmpty(msg.content) ? "保存成功" : msg.content)
                               + "，但桌名热更新失败：" + (tc == null ? "服务端无响应" : tc.content);
                }
            }
            catch (Exception exTc)
            {
                LogHelper.WriteLog(typeof(B_LabaGamePara), exTc);
                msg.datas = true;
                msg.content = (string.IsNullOrEmpty(msg.content) ? "保存成功" : msg.content)
                           + "，但桌名热更新异常：" + exTc.Message;
            }
        }
    }
    return msg;
}
```

**关键修复点:**
1. **TC 解耦门控**:移出 `if (val > 0)`,只要 tableName 非空就发(修复改名失效)
2. **upsert 替代 update**:找不到 OptKey 就插入(修复 ExchangeScore update 空转)
3. **新增 enabled 参数**:传入桌台开关,不再固定写 1
4. **删除 winKey/winRate 逻辑**:不再构造 winRate 相关 paras

### 5.4 数据库改动

无需加表加列,复用现有:
- `gameconfiglaba` 存 Payout0-9 / betMin / betMax / coinsNeed / defaultBetIndex(OptKey/OptValue 键值对,天然支持任意字段)
- `roomtableconfig` 存桌名 / Enabled(经 TC 命令)
- `parabet` 存 Shz 的 dif/har(Shz 走押注表,已有)

> dif/har 保存:Shz 的 dif/har 在 `ParaBet` 表(DIF/HAR 列),保存时需 upsert ParaBet 行(参考 gameType==0 押注保存的 `M_ParaBet` 构造)。
