## 二、逐项审计明细

### 需求 1 — 动态桌台/机台配置（✅ 已完成）
- 落地范围见 `02-动态桌台机台配置-实现进度.md`：统一配置页 `ConfigEditor.cshtml`、`GameConfigController`（增删改/校验/单位换算/按类型路由 BLL）、`B_SuperPara` 命令式热更新强校验（先 Commit 再 `EScMsgType.RP`，失败不静默）、`pararoom/parabetroom` 扩展列（TableName/MaxSeats/IdleFire*/Enabled + MinBetUnits/MaxBetUnits）。
- 近期已进一步简化「无发炮踢出」：移除开关 toggle，「无发炮踢出时间（秒）」默认开启并常驻显示（`IdleFireKickEnabled` 隐藏域固定提交 1，控制器默认值改为 1）。
- 结论：后台侧闭环。剩余仅 C++ 端按 `MaxSeats`/`scale=10`/`IdleFire*` 对齐数组扩容与读取（跨端联调，已在 02 文档「下一步」列明）。

### 需求 2 — 鱼机炮值与难度（🟡 后台雏形，算法待服务端）
- 已具备：下注炮值的整数缩放通道（`MinBetUnits/MaxBetUnits = round(显示值×10)`），避免浮点贯穿 DB/协议。
- 缺口：
  - 0.5 粒度难度（`parafish.DIF` 当前整数）、小/大倍率鱼死亡率、出鱼数量——均属**子游戏算法**，不是后台改字段能闭环的。
  - 后台侧仅需：放开小数输入框、固化「显示值↔协议值 scale=10」换算文案、难度按 0.5 粒度入库为整数（×2 或 ×10）。
- 结论：后台工作量小且与需求 1 同源；**主导方是服务端算法**，建议并入「鱼机/牌机算法联调」由服务端牵头。

### 需求 3 — 牌机出货/库存/新增游戏（🔴 未实现）
- 现状：`cardpayoutprofile` 表已在 `mth.sql`（建表于约 221 行）但**代码零引用**；`paracard.DIF` 仍是 `3434453555777533` 这类不透明字符串；牌型倍数/概率/库存无结构化表与可视化配置。
- 缺口（后台）：新增/启用 `cardpayoutprofile`（GameId/TableId/HandType/PayoutMultiplier/ProbabilityBasis/StockLimit/StockRemain/Enabled）实体与 BLL、每台机×每牌型的编辑 UI、旧 `DIF` 兼容映射。
- 跨端缺口：新算法需服务端读取新表并按牌型出货；新增游戏（ATT/水浒/明星97/水果机）必须确认服务端项目枚举与客户端资源到位，不能只改 `games`。
- 结论：后台中高风险（新表+新 UI+迁移），且强依赖服务端算法落地，排在需求 4 之后。

### 需求 4 — 玩家控制：点杀/放水/金币上限（🔴 状态机未实现）
- 现状：`usercontrolstatus`、`usercontrolvalue` 表已建（`mth.sql`）但**代码未接入**；现有 `ChgUserRate`（鱼机）/`ChgUserRateBet`（押注）/`ChgUserRatePAIJI`（牌机）是旧「概率值/次数」控制；`manageropt` 仅作一次性记录。
- 缺口（后台）：
  - 控制状态机：`ControlMode(Kill/Release/Limit)` + `TargetCoins/ConsumedCoins/GrantedCoins/LimitCoins` + `Status` + `CreatedBy/Time/ExpiredTime`，达标自动失效、服务端重启可恢复。
  - 权限拆分：牌机「控制」与「放水」分权；后台「匿名控制」改名「放水」。
  - 后台备注/进度展示：代理可直观看到某玩家当前控制模式、目标、已吃/已放进度。
- 跨端缺口：控制要真正生效，`SConnect.SendReadString(EScMsgType.UC, ...)` 的命令内容需服务端支持扩展参数（不能只传旧格式）。
- 结论：**最高业务价值**，且 DB schema 已预置（信号是「本就计划的下一主线」）。后台可独立交付「状态表+状态机+权限+UI+备注进度」，但「慢慢吃/慢慢给」的实际执行需服务端 UC 协议扩展，须同步立项。

### 需求 5 — 在线用户实时监管（🟡 有基线，待增强）
- 已具备：`UserInfoController.GetOnlinePlayers` → `B_Users.GetOnlinePlayersList` → `FillCurrentPlayInfo` 填充当前游戏/桌台/下注/输赢；视图 `OnlinePlayers.cshtml`（另有 `OnlineCoin.cshtml`）；`M_Users_DTO` 已有 `CurrentGameName/CurrentRoomId/CurrentBetCoins` 等实时字段；甚至有视图契约单测。
- 缺口：
  - 聚合列：总充值 / 总兑换 / 总盈亏 / 游戏分尚未在列表聚合。
  - 控制可视化：控制备注、当前控制模式与进度（与需求 4 联动）。
  - 「仅游戏内、不显示大厅」当前以 `INHALL` 表达，语义需细分（在线 / 在大厅 / 在房间 / 在游戏中）。
  - 实时刷新来源需确认（服务端持续上报 vs 查旧日志），第一阶段可用短轮询，SignalR 已引入但不强依赖。
  - 权限过滤：代理仅可见自身管理范围（沿用 `B_Admin.GetManagedAgencyAccounts()` 能力）。
- 结论：**纯后台可交付、风险低、见效快**；是需求 4 控制操作的「可视化前置」。

### 需求 6 — 代理客服链接（🟢 基本完成）
- 已具备：`admin.CustomerServiceUrl` 字段、`B_Admin.ChgCustomerServiceUrl`、后台编辑入口 `AgencyInfoController.ChgCustomerServiceUrl`、前台解析 `Api/CustomerServiceUrlController`（`GetSingle(Users.AGENCY) → CustomerServiceUrl`）、注册时 `InviteCode → AGENCY` 绑定。
- 缺口：下级代理客服链接为空时的回退规则（向上查找 / 总后台默认）未固定；编辑权限约束待补。
- 结论：低成本收尾，建议与需求 5 同期顺手清掉。

### 需求 7 — 异常断线/资金强一致性（🔴 架构级，跨端专项）
- 现状：`userscoresnapshotledger` 表存在；核心结算逻辑在 C++ 中心服，后台无金币增量事务结算（仓内 `Gold = Gold + x` 仅见于 `UserRobotController` 机器人管理，非玩家核心结算）。
- 本质：高并发卡机/断线/顶号导致的吞分，是 C++ 内存状态与 MySQL 事务同步问题，**单靠后台仓库无法解决**。
- 建议（双重防护网，须服务端牵头）：
  - C++：心跳/断线托管，异常断线即触发结算异步落库；顶号「踢 A → 强制结算写库并等确认 → 放行 B」互斥三步走；内存赢余秒级快照（可选 Redis 缓存层）。
  - MySQL：结算包裹完整事务（余额表与流水表同生共死），**强制增量更新 + 行锁，禁止绝对值覆盖**。
- 结论：需单独立项、跨端协作；后台侧仅能配合做流水/快照核对与对账页。**强提醒服务端 C++ 同步座位数组长度与结算一致性，避免硬编码 `[4]` 越界与本金蒸发。**

## 三、下一步推荐

**推荐主线顺序：需求 5（在线用户监管增强）→ 需求 4（玩家控制状态机）→ 需求 3（牌机结构化出货），需求 6 随手收尾，需求 2/7 标记为服务端主导/专项。**

### 第一步：需求 5「在线用户实时监管」增强（建议立即开工）
理由：
1. **纯后台可交付、零 C++ 强依赖**，已有可用基线（控制器/BLL/视图/DTO 齐备），属增强而非新建，风险最低、交付最快。
2. 契合总览文档（01）自身的「后台页面顺序：桌台 → 在线用户 → 控制弹窗 → 代理客服」。
3. 是需求 4 控制操作的**可视化前置**：先让代理「看得见在玩的人」，才能高效「点杀/放水」。
4. 立即产生运营价值（实时余额/下注/盈亏/聚合）。

落地要点（下一阶段实现时）：在 `M_Users_DTO` 扩聚合字段（总充值/兑换/盈亏/游戏分）+ 列表列；细化在线状态（在线/大厅/房间/游戏中）严格「仅游戏内」过滤；短轮询刷新；代理范围权限过滤；预留控制备注/进度列对接需求 4。

### 第二步：需求 4「玩家控制状态机」（最高业务价值，紧随其后）
- 后台先落「`usercontrolstatus` 状态机 + 权限拆分（牌机控制/放水、匿名控制→放水改名）+ 后台 UI + 备注进度」，并接入需求 5 的在线列表。
- **同步推动服务端扩展 `EScMsgType.UC` 命令参数**（目标金额/慢吃慢给/自动失效），否则后台状态机无法真正生效。

### 收尾与专项
- **需求 6**：补「客服链接回退规则 + 编辑权限」，与需求 5 同期收尾。
- **需求 2**：后台放开小数输入与 scale 文案即可，难度/出鱼算法随**服务端**「鱼机算法联调」推进。
- **需求 3**：在需求 4 之后启动，需新表+UI+服务端算法绑定+客户端资源确认，避免与服务端脱节。
- **需求 7**：单独立项、跨端联合改造，后台仅配合对账；**最高架构风险，须优先排期评估**。

## 四、重点风险与依赖
- 只改后台表单、服务端协议/子游戏内存不更新，前台仍不会实时生效（需求 2/3/4 通病）。
- 小数炮值/难度坚持整数缩放贯穿 DB/C++/客户端，杜绝浮点误差。
- 玩家控制必须做成可恢复状态机，否则服务端重启后「吃了多少/放了多少」丢失。
- 在线用户页若依赖日志而非当前状态，会延迟、重复、误显示大厅用户。
- 牌机结构化出货须保留旧 `paracard.DIF` 兼容，否则旧游戏无法启动。
- 坐席 4→6 扩容须强提醒 C++ 同步座位数组长度，防内存越界崩溃。
- 资金强一致性是最致命架构风险，必须「C++ 状态守护 + MySQL 增量事务」联合落实。
