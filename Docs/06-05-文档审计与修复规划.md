## 二、代码层面潜在问题（文档未覆盖）

### P0 — 严重：`CustomerServiceUrlController` 缺少 null 保护

**文件：** `TTY.Web/Controllers/Api/CustomerServiceUrlController.cs:28`

```csharp
string playerId = model.playerId.Trim();  // model 可能为 null
```

**场景：** 前端 POST body 为空或格式错误时，`model` 反序列化结果为 `null`，`model.playerId` 直接抛出 `NullReferenceException`。虽然外层有 `try-catch`，但异常会被转为 500 错误响应而非业务错误码。

**修复方案：**

```csharp
if (model == null || string.IsNullOrWhiteSpace(model.playerId))
{
    msg1.content = "用户账号不能为空！";
    return msg1;
}
string playerId = model.playerId.Trim();
```

### P1 — 中等：`lock(this)` 在 Web API 中并发保护无效

**文件：** `TTY.Web/Controllers/Api/CustomerServiceUrlController.cs:23`

```csharp
lock (this) { ... }
```

**原因：** ASP.NET Web API 默认每个请求创建新的 Controller 实例。`lock(this)` 锁的是不同对象实例，无法实现跨请求互斥。即使存在实例复用缓存，也不保证行为一致。

**影响：** 当前接口逻辑简单（单次 DB 查询 + 返回），无状态写入，实际风险较低。但 `lock(this)` 传递了"这是线程安全代码"的错误信号，应明确标注或删除。

**修复方案：** 二选一：

1. **删除 lock**（推荐）：确认无并发写入需求后直接移除，加注释说明为何不需要锁。
2. **改用静态锁**：`private static readonly object _lock = new object(); lock (_lock) { ... }`，但需确认确实有并发保护必要。

### P2 — 低优：`fmtProfit` 对 `0` 值的 `||` 语义不精确

**文件：** `TTY.Web/Areas/Game/Views/UserInfo/OnlinePlayers.cshtml:84-88`

```javascript
function fmtProfit(value, rowData, rowIndex) {
    var v = value || 0;  // value 为 0 时，0 || 0 → 0，功能正确但语义不对
    ...
}
```

**说明：** `value` 类型为 `long?`（nullable），当值为 `0` 时 `0 || 0` 结果虽正确，但应使用 `??`（null-coalescing）更准确地表达意图。

**修复方案：**

```javascript
var v = value ?? 0;
```

---

### P3 — 低优：自动刷新 DOM 查询脆弱

**文件：** `TTY.Web/Areas/Game/Views/UserInfo/OnlinePlayers.cshtml:93-94`

```javascript
if (document.getElementById('chkAutoRefresh').checked) {
    var sec = parseInt(document.getElementById('selRefreshSec').value, 10) || 10;
```

**说明：** 使用原生 `document.getElementById` 而非 jQuery（项目整体使用 EasyUI/jQuery），与其他代码风格不一致。且若在 DOM 渲染前调用会静默失败。

**修复方案：** 统一使用 jQuery 选择器：

```javascript
if ($('#chkAutoRefresh').is(':checked')) {
    var sec = parseInt($('#selRefreshSec').val(), 10) || 10;
```
