# 03·数据库脏数据清洗——幽灵桌台与失效配置归档

> 本轮为"动态桌台删除"全链路开发的**前置数据治理**：在编译/热更部署前，先把历史
> `GAME_ID*1000+i` 硬编码遗留的幽灵桌台、超额房间、孤儿游戏等脏配置识别并归档清洗脚本，
> 避免脏数据导致后续删除逻辑与引擎内存重建错位。**本轮只产出脚本，不在库上执行 DELETE。**

## 一·问题根因与核心/非核心数据区分

### 1.1 引擎数据模型（判定脏数据的唯一基准）
中心服 `ServerCenterNew/Database.cpp` 的 `GetFishPara / GetCardPara / GetBetPara` 对每个游戏：

- 固定 `ROOM_MAX = 3`，只按 `ID = GAME_ID*1000 + (0/1/2)` 读取 **3 个房间**（pararoom/parabetroom）；
- 真实桌台数 `tableMax = 房间0.NUM + 房间1.NUM + 房间2.NUM`；
- 只按 `ID = GAME_ID*1000 + (0 .. tableMax-1)` 读取桌台行（parafish/paracard/parabet）。

因此凡是**房间偏移 ≥ 3**、**桌台偏移 ≥ tableMax**、或**游戏不在 games 主表**的 para* 行，
引擎一律不加载，属于旧版硬编码遗留的"幽灵数据"，只会扰乱后台列表与统计。

### 1.2 核心数据白名单（绝对隔离、本轮零触碰）
以下表承载玩家账号、资金流水、权限与全局状态，**任何清洗脚本都不得涉及**：

`users`、`accountlog`、`safe_coins_log`、`admin`、`gamemo`、`useroptlog`、
`banklog`、`rechargelog`、`exchangelog` 等一切用户/资金/日志/权限表。

本轮所有 DELETE 仅落在 **para\*** 业务配置表（`parafish` / `pararoom` / `parabetroom` /
`paracard` / `paragame`），这些表的特征是"可由后台重新配置 + 引擎启动时重建内存"，
属于可重建的业务参数，与核心数据严格区分。

## 三·改动内容（在 `Docs/sql/` 下新增 4 份清洗脚本）

均为纯中文注释、UTF-8(No BOM)，结构统一：**第一步只读 SELECT 核对条数 → 第二步
`START TRANSACTION` 包裹的带注释 DELETE → `COMMIT`（异常改 ROLLBACK）**。

1. `清理失效鱼机桌台参数.txt` —— parafish 超出 tableMax 的桌台行（186 行）。
2. `清理失效鱼机房间配置.txt` —— pararoom 房间偏移 ≥ 3 的房间行（4 行）。
3. `清理失效押注房间配置.txt` —— parabetroom 房间偏移 ≥ 3 的房间行（1 行）。
4. `清理幽灵游戏配置.txt` —— games 主表不存在的游戏 51/52 在 para* 的全部配置（53 行）。

本轮**未改动**任何 C#/C++/前端代码，未改表结构，未执行任何 DELETE。

## 四·关联风险预警

1. **误删核心数据（已规避）**：所有 WHERE 均限定在 para* 表且带 `GAME_ID` / 偏移上界，
   不含任何用户/资金表；脚本强制"先 SELECT 核对再 DELETE"，杜绝整表误删。
2. **后台重建兼容性**：删除幽灵房间后，后台桌台配置页（按 pararoom 行展示）对游戏 19/49
   显示的房间数会从 6/4 减为 3，与引擎一致；**但需注意**当前后台保存逻辑会把 `NUM` 回写为
   "房间行数"（见 02 文档三·十），属于与引擎"NUM=每房间桌台数"语义冲突的历史问题——
   在"动态桌台删除（方案 A）"落地前，**操作者不应对已清洗游戏触发整表 NUM 重算保存**，
   否则会把 tableMax 改小、与 parafish 行数错位。该语义统一将在下一阶段处理。
3. **执行顺序**：建议先执行幽灵游戏与房间脚本，再执行桌台脚本；每份脚本独立事务，
   核对条数与本文一致后再 COMMIT。
4. **备份**：执行前对 para* 配置表做一次逻辑备份（mysqldump 仅导出这几张表），留回退底牌。

## 五·下一步处理建议

1. 由 DBA/运维按"先 SELECT 核对 → 条数吻合 → COMMIT"流程，在停服或低峰期执行 4 份脚本。
2. 清洗完成后，配置表即满足引擎"3 房 + 连续 ID"模型，可安全推进**动态桌台删除（方案 A）**：
   C# 先校验在线玩家再物理删库 + id 连续重排 + 重算 NUM/tableMax，RP 热更触发 C++ 整表
   加写锁重建 roomInfo/tablePara/gameTable。
3. 长期：统一后台与引擎对 `NUM` 的语义（消除"行数 vs 每房间桌数"分歧），从源头杜绝幽灵数据再生。
