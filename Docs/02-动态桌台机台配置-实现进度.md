# 动态桌台配置改造 — 实现进度

本文记录“大厅 → 进入游戏 → 平铺展示独立桌台”改造的后台落地情况，便于查看进度与下次续接。基于 `01-改造总览与需求审查.md` 的方案推进。

## 一、问题根因

1. 旧结构以“房间模板（pararoom）+ 机台（parafish/paracard）”分层，前台呈现“初/中/高级”分段，无法满足“每张桌台独立配置”。
2. `pararoom` 缺少桌名、坐席数、无发炮踢出等字段，最大坐席固定 4 人。
3. 下注精度为整数，业务需要 0.1 起步，但数据库与 C++ 协议不能用浮点。
4. 保存后是否真正热更新缺少强校验，存在“写库成功但服务端未刷新”的静默失败风险。

## 二、本次改动内容

### 1. EF6 模型扩展 — `YYT.Entity/EF/M_ParaRoom.cs`
在不破坏旧字段的前提下新增（均为新增列，旧逻辑不受影响）：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `TableName` | string | 桌名 |
| `MaxSeats` | int，默认 6 | 最大坐席数，配合 C++ 端数组扩容防越界 |
| `IdleFireTimeoutSec` | int | 无发炮踢出时间（秒） |
| `IdleFireKickEnabled` | bool/bit | 无发炮踢出开关 |
| `Enabled` | bool/bit，默认 true | 桌台开关 |
| `MinBetUnits` / `MaxBetUnits` | int | 最小/最大下注炮值（整数单位，显示值放大 10 倍入库） |

### 4. 命令式热更新强校验 — `YYT.BLL/EF/shit/B_SuperPara.cs`（新增 `SaveTableConfig`）

## 三、单位换算约定（务必前后端一致）
- 前台显示值支持 0.1，入库 `MinBetUnits/MaxBetUnits = round(显示值 × 10)`。
- 读取展示时 `显示值 = Units / 10`；旧数据无 Units 时回退 `BET_MIN × 10` 再 ÷10。
- 数据库与协议层只走整数，C++ 端按 `scale=10` 解释。

## 三·九 桌台列表自动刷新 / 参数预填快速建台 / 重置归零 / 空态占位符（本次）
针对反馈的 4 点交互问题修复，集中在 `ConfigEditor.cshtml` 与 `B_SuperPara.cs`：

1. 桌台列表保存后不自动刷新（根因）：`PushHotUpdate` 在 C++ 中心服离线时返回 `code=0`（房间热更新失败），但此时数据库已 Commit。前端原逻辑仅在 `code==1` 才刷新列表，导致新建/更新成功但列表不更新。
   - 修复：`B_SuperPara.PushHotUpdate` 在“写库成功、热更新失败”分支额外置 `result.datas = true` 作为“已落库”标记（仍保留明确的失败提示文案）。
   - 前端 `gcSave` 改为 `data.code == 1 || data.datas === true` 即刷新列表（`gcLoadTables` 从数据库读取，与热更新状态无关）；数据库本身失败（catch 分支 `code=0` 且无 `datas`）则不刷新。
2. 参数预填快速建台：`gcNewTable` 改为自动套用该游戏“最近创建桌台”（`gcLatestTable`，取 `id` 最大者）的全部参数，仅桌名 `TableName` 留空；该游戏暂无桌台时回退为归零并默认启用、坐席 6。三类游戏统一生效。选中游戏或保存成功后默认进入此“预填新建态”。
3. 重置按钮：`gcReset` 改绑 `gcZeroForm`，将全部数值输入归零、文本清空、坐席复位 6、开关取消勾选、筹码清空（保留当前游戏上下文 `ID/GAME_ID/GAME_TYPE`）。
4. 空态占位符：未选择游戏类型/游戏时，右侧参数区隐藏整张表单与底部按钮，改为居中、带图标(`fa-gamepad`)的优雅占位提示（`#gcParaEmpty`），由 `gcSetParaVisible` 统一切换。

辅助：`gcLoadTables(afterLoad)` 支持加载完成回调；列表渲染抽出 `gcRenderTableList`，按 `id` 选中抽出 `gcSelectTableById`（保存为更新时回选该桌台，新建时进入预填新建态）。
